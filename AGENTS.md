# Clawmail Development Guide

Agent-oriented email system: domain registration, inbox API keys, SMTP inbound, REST API for read/send/reply/archive.

## Quick Start

```bash
cp .env.example .env          # edit DATABASE_URL, DKIM_ENCRYPTION_KEY (64 hex chars)
pnpm install
pnpm run migrate              # creates all tables in PostgreSQL
pnpm run dev                  # Express on PORT (default 3000), SMTP on SMTP_PORT (default 25)
```

## Commands

- `pnpm start` — production server
- `pnpm run dev` — development with nodemon auto-restart
- `pnpm run migrate` — run SQL migrations (`node src/db/migrate.js`)

  middleware/
    domainAuth.js       # Bearer <verification_token> → req.verifiedDomain {id, domain}

    dkim.js             # RSA-2048 keygen, AES-256-GCM encrypt/decrypt, DNS record builder
  services/

    inboxService.js     # findOrCreateInbox, listInboxes, getInboxByAddress
    apiKeyService.js    # createApiKey, validateApiKey
    emailService.js     # listEmails, getEmail, sendNewEmail, replyToEmail, archiveEmail

    errors.js           # AppError, NotFoundError, BadRequestError, UnauthorizedError, ConflictError

## Architecture Patterns

### Auth
Two auth schemes, both via `Authorization: Bearer` header:
- **Domain-level** (`domainAuth.js`): plain verification_token → sets `req.verifiedDomain`
- **Inbox-level** (`auth.js`): `email:api_key` format → sets `req.inbox`

Domain auth is distinguished from inbox auth by the absence of a colon in the token.

### Database
- All queries use parameterized `db.query(text, params)` — never interpolate user input
- For transactions, use `db.getClient()` then `BEGIN`/`COMMIT`/`ROLLBACK` with `client.release()` in finally
- UUIDs generated by PostgreSQL (`uuid_generate_v4()`) for PKs
- JSONB columns for to/cc/bcc address arrays and headers

### Route Pattern
Every route handler follows: `async (req, res, next) => { try { ... } catch (err) { next(err); } }`. Throw `AppError` subclasses for operational errors; unhandled errors become 500s.

### Rate Limits
- Domain registration: 5/hour
- Inbox email send/reply: 10/hour
- Inbox email reads/archive: 100/hour

Inbox email rate limits are applied per route in `routes/emails.js`. They default to client-IP buckets and can be switched to per-inbox-email buckets with `INBOX_RATE_LIMIT_SCOPE=email`.

### Email Threading
Thread IDs computed by looking up `In-Reply-To` or `References` headers against existing `message_id` values. New threads get a fresh UUID.

### SMTP Inbound Flow
1. `onRcptTo`: reject if domain not in `verified_domains`
2. `onData`: parse with `mailparser.simpleParser`, for each recipient on a verified domain:
   - Find-or-create inbox (auto-creates unknown local parts)
   - Compute thread_id from In-Reply-To/References
   - Insert into `emails`, save attachments to `ATTACHMENTS_DIR/<email_id>/`
   - Process forwarding rules (fire-and-forget, errors logged not thrown)

### Outbound
- Nodemailer `direct: true` transport (resolves recipient MX, no relay)
- DKIM signed with per-domain RSA-2048 keys (decrypted from DB at send time)
- Outbound copy stored in `emails` with `direction='outbound'`, `is_read=true`

### DKIM Key Storage
Private keys encrypted at rest with AES-256-GCM using `DKIM_ENCRYPTION_KEY` env var (must be 64 hex chars = 32 bytes). IV and auth tag are prepended to ciphertext and base64-encoded.

## Adding New Migrations

Create `src/migrations/002_<name>.sql`. The runner applies files in sorted order and tracks applied migrations in a `migrations` table. Each migration runs in a transaction.

## Adding a New Endpoint

1. Add service function in `src/services/`
2. Add route in `src/routes/`, using the standard try/catch/next pattern
3. Mount in `src/index.js` with appropriate rate limiter
4. Use `domainAuth` or `inboxAuth` middleware as needed
5. Document the external contract in `API.md` and update `SKILLS.md` when agent workflows or operational guidance change

## API Keys
Format: `cm_<64 hex chars>`. Only SHA-256 hash stored in DB. Plaintext returned once on creation. `key_prefix` (first 12 chars) stored for identification. Keys can be revoked via `revoked_at` timestamp.

## Environment Variables

| Variable | Description | Default |
|---|---|---|
| `PORT` | Express HTTP port | 3000 |
| `TRUST_PROXY` | Express trust proxy setting used for client IP handling behind a reverse proxy | false |
| `INBOX_RATE_LIMIT_SCOPE` | Inbox route limiter key: `ip` or `email` | ip |
| `DATABASE_URL` | PostgreSQL connection string | `postgresql://clawmail:clawmail@localhost:5432/clawmail` |
| `DKIM_ENCRYPTION_KEY` | 64 hex char key for DKIM private key encryption | (required) |
| `SERVER_IP` | Expected IP for MX/A record verification | 203.0.113.10 |
| `SMTP_BANNER_HOSTNAME` | SMTP EHLO hostname (should match rDNS PTR) | derived from `APP_DOMAIN` as `mx.<APP_DOMAIN>` when set |
| `SMTP_PORT` | Inbound SMTP port | 25 |
| `LOG_LEVEL` | Pino log level | info |
| `NODE_ENV` | Environment (enables pino-pretty in development) | development |
| `ATTACHMENTS_DIR` | Directory for email attachment files | ./attachments |
